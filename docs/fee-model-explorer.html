How the Fee Model Works

The Zero-Coupon Bond Analogy

When the Ark operator creates a VTXO, it locks its own BTC as a counterparty guarantee for the duration of the VTXO's lifetime. This is economically equivalent to buying a zero-coupon bond: the operator lends capital now and gets it back when the VTXO expires. The liquidity fee compensates the operator for the time-value of that locked capital, just as a bond's discount compensates the lender.

Liquidity Cost is Proportional to Amount × Time

The liquidity component A · (δ/365) · r scales linearly with both the VTXO value and the remaining time the operator's capital stays locked. A 1 BTC VTXO with 30 days remaining costs exactly 10× more in liquidity fees than a 0.1 BTC VTXO with the same lifetime, and twice as much as a 1 BTC VTXO with 15 days remaining.

Batch Size Reduces On-chain Cost, Not Liquidity Cost

The on-chain share Fround/B shrinks as more participants join a round, because the fixed transaction overhead is amortized. However, the liquidity fee depends only on each user's own amount and time — adding more participants to the round does not reduce it. For large VTXOs, liquidity dominates; batching helps most for small VTXOs where on-chain cost is the main component.

Annual Cost Insight

The "Annual Cost of Ownership" chart assumes the user refreshes at half-lifetime (δ = L/2), which is the cadence that makes the annualized liquidity component equal to A · r. At this cadence, a 4-week timelock means ~26 refreshes/year paying 2 weeks of interest each, and a 1-week timelock means ~104 refreshes/year paying half a week of interest each — both liquidity totals equal A · r. The visible difference between the curves comes from the on-chain share and base margin, which are paid per round and thus favor longer timelocks (fewer rounds per year).

Refreshing earlier (larger δ) increases the annualized liquidity cost above A · r; refreshing lazily (smaller δ, closer to expiry) decreases it. The precise closed-form is C_liq_annual = A · r · x / (L − x) where x is the remaining lifetime at refresh.

OOR Transfers are Free

Off-round (OOR) transfers between users already inside the Ark do not require the operator to deploy new capital — the existing VTXO simply changes hands. Because no new liquidity is locked, the operator bears no time-value cost, and OOR transfers can be offered at zero or near-zero fees.

Boarding is Free of Liquidity Fee

Boarding brings on-chain BTC from the user into the shared output. The operator's own wallet is unchanged and no new operator capital is locked up, so boarding pays only the on-chain share and flat margin — no liquidity fee. The double-commitment rationale that prices refresh has no analog in boarding (there is no prior UTXO the operator is waiting to recover). See fee-model.md "Boarding vs Refresh" and the tracking issue on this decision for the full argument.

The charts above use the refresh/offboard pricing (with liquidity fee). For the boarding case, set the liquidity term to zero mentally — the total collapses to F_round/B + ε.

Refresh Fee Floor (δ_min)

To prevent a "lazy refresh" bypass — refreshing at δ ≈ 0 just before expiry for a near-zero liquidity fee — the refresh fee uses a floor on the effective δ: liquidity_fee = A · max(δ, δ_min) / 365 · r_eff. The default δ_min is ~1 day. Users can still refresh at any δ — the floor only affects the fee calculation, not admission — so no one is forced into unilateral on-chain exit. The breakdown panel shows "floor" when the floor is active for your current parameter set.

Exit Cost and Tree Structure

Each VTXO sits as a leaf in a Merkle-like tree. To unilaterally exit, a user must reveal every branch from their leaf to the root on-chain. The tree's radix (branching factor R) determines depth: depth = ceil(log_R(B)). Each branch transaction must reveal R−1 sibling hashes as witness data.

This creates a tradeoff: low radix (R=2) means many levels with small branch transactions; high radix (R=8) means few levels but large branch transactions. The "Exit Cost vs Radix" chart shows the resulting U-shaped cost curve. The optimal radix minimizes total exit cost for a given batch size and feerate.

The effective minimum viable VTXO is the maximum of the economic dust threshold (where fees exceed 50% of value) and the security dust threshold (where exit cost exceeds the VTXO value). For small VTXOs at high feerates, exit cost is typically the binding constraint.

Disclaimer: the branch and claim transaction sizes used in the exit cost formula (branchVBytes(R) = 80 + (R−1)·32, claimVBytes = 150) are ballpark P2TR estimates for illustration. The real implementation must compute these from the actual transaction structure; treat the exit-cost curves here as shape-accurate and magnitude-approximate.

Congestion Spread for Operators

The effective rate r_eff = r + Δ(u) rises when the operator's treasury utilization u exceeds a configurable threshold u*. Below the threshold, Δ = 0 and users pay only the base rate. Above it, Δ = Δ0 + Δ1 × (u − u*), where Δ0 is a base spread and Δ1 is a linear coefficient.

In practice: if an operator's treasury is 85% deployed (u = 0.85) with a threshold of 70% (u* = 0.70), the congestion spread automatically raises rates to discourage further capital deployment until sweeps and VTXO expirations return funds. Operators tune u* to set their comfort level, and Δ0/Δ1 to control how aggressively rates climb above that level.
