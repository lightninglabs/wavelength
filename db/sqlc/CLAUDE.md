# db/sqlc

## Purpose

Type-safe, generated SQL query layer for the main client schema. Generated by
`sqlc` from the hand-written statements in `db/sqlc/queries/*.sql` against the
schema in `db/sqlc/schemas/generated_schema.sql`; the `db` package builds its
stores on top of the `Queries` type and `Querier` interface this package
exposes instead of writing SQL directly.

## Key Types

- `Querier` — Interface covering every generated query method (boarding,
  rounds, VTXOs, ledger/fee accounting, OOR artifacts and session registry,
  pending intents, spending reservations, unilateral exit jobs, UTXO audit
  log, vHTLC recovery jobs, credit operations, internal keys, chain info).
  `Queries` implements it; `db` stores depend on this interface, not the
  concrete type, so tests can substitute fakes.
- `Queries` / `DBTX` — `Queries` wraps a `DBTX` (either `*sql.DB` or `*sql.Tx`)
  and is the concrete implementation of `Querier`. `WithTx(tx)` rebinds it to
  a transaction.
- `BackendType` (`db_custom.go`, hand-written, not sqlc output) —
  `BackendTypeSqlite` / `BackendTypePostgres` / `BackendTypeUnknown`.
  `NewSqlite(db)` / `NewPostgres(db)` construct a `Queries` tagged with its
  backend so `(*Queries).Backend()` can be queried later (used by
  `db/migrate` to pick the right migration driver).
- Row/model types in `models.go` — one struct per table, e.g. `Round`,
  `Vtxo`, `VtxoAncestryPath`, `BoardingIntent`, `BoardingSweep`,
  `LedgerEntry`, `CreditOperation`, `OorSessionRegistry`, `PendingIntent`,
  `PendingIntentAnchor`, `SpendingReservation`, `UnilateralExitJob`,
  `VhtlcRecoveryJob`, `InternalKey`, `WalletUtxoLog`.
- Per-table `*.sql.go` files (`round.sql.go`, `vtxo.sql.go`,
  `boarding.sql.go`, `credit_operations.sql.go`,
  `oor_session_registry.sql.go`, `pending_intents.sql.go`,
  `spending_reservations.sql.go`, `unilateral_exit.sql.go`,
  `utxo_audit.sql.go`, `vhtlc_recovery_jobs.sql.go`, `internal_keys.sql.go`,
  `fee_accounting.sql.go`, `chain_info.sql.go`, `oor_artifacts.sql.go`) —
  each generated 1:1 from the matching file in `db/sqlc/queries/`.

## Relationships

- **Depends on**: `database/sql` only; no dependency on other darepo-client
  packages (keeps the generated layer a leaf so it can be regenerated freely).
- **Depended on by**: `db` (builds all higher-level stores on `Querier`),
  `db/migrate` (`BackendType` selects the migration driver).

## Invariants

- **Generated code — never hand-edit.** Regenerate via `make sqlc` after
  changing `db/sqlc/queries/*.sql` or the schema. The `*.sql.go`, `models.go`,
  `querier.go`, and `db.go` files all carry a `// Code generated by sqlc. DO
  NOT EDIT.` header.
- `db_custom.go` is the one hand-written file in this package (no generated
  header) — it exists because sqlc has no way to express backend-tagging
  constructors; keep hand-written additions confined to that file.
- To add or change a query: edit the matching `.sql` file under
  `db/sqlc/queries/`, update `db/sqlc/schemas/generated_schema.sql` if the
  table shape changed (via a migration, not a direct schema edit), then run
  `make sqlc`. Never write raw SQL inline in `db` package Go code.
- `db/sqlc/migrations` holds the raw SQL migration files golang-migrate
  applies; `db/migrate` and `db.migrations.go` drive them, this package does
  not run migrations itself.
