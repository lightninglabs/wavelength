# db/actordelivery/sqlc

## Purpose

Type-safe, generated SQL query layer for the durable actor mailbox schema.
Generated by `sqlc` from `db/actordelivery/queries/mailbox.sql` against the
actor-delivery migrations; `db/actordelivery` builds the durable mailbox,
outbox, ask-result, dead-letter, and FSM-checkpoint stores on top of the
`Queries` type and `Querier` interface this package exposes instead of
writing SQL directly.

## Key Types

- `Querier` — Interface covering every generated mailbox/outbox query method:
  enqueue/lease/peek/ack/nack/extend-lease/expire for the durable mailbox,
  claim/complete/fail for the transactional outbox (CDC pattern),
  insert/get/delete for ask results, insert/list for FSM checkpoints,
  move/list/delete for dead letters, and mark/check for processed-message
  deduplication. `Queries` implements it; `db/actordelivery` stores depend on
  this interface so tests can substitute fakes.
- `Queries` / `DBTX` — `Queries` wraps a `DBTX` (either `*sql.DB` or `*sql.Tx`)
  and is the concrete implementation of `Querier`. `WithTx(tx)` rebinds it to
  a transaction.
- Row/model types in `models.go` — `MailboxMessage` (durable inbox row, one
  per pending/leased message), `OutboxMessage` (transactional outbox row for
  the CDC delivery pattern), `AskResult` (stored reply for a completed Ask),
  `FsmCheckpoint` (durable actor FSM state snapshot), `DeadLetter` (poison
  message moved out of the mailbox/outbox after exhausting `max_attempts`),
  `ProcessedMessage` (deduplication ledger entry).
- `mailbox.sql.go` — the single generated query file, produced 1:1 from
  `db/actordelivery/queries/mailbox.sql`.

## Relationships

- **Depends on**: `database/sql` only; no dependency on other
  darepo-client packages, keeping the generated layer a leaf.
- **Depended on by**: `db/actordelivery` (`Store`, `TxActorDeliveryStore`,
  `TxAwareActorDeliveryStore` all wrap `Querier`/`Queries` as their
  `ActorDeliveryQueries` implementation).

## Invariants

- **Generated code — never hand-edit.** Regenerate via `make sqlc` after
  changing `db/actordelivery/queries/mailbox.sql` or the actor-delivery
  schema/migrations. `mailbox.sql.go`, `models.go`, `querier.go`, and `db.go`
  all carry a `// Code generated by sqlc. DO NOT EDIT.` header.
- To add or change a mailbox query: edit
  `db/actordelivery/queries/mailbox.sql`, add/adjust the corresponding
  migration under `db/actordelivery/migrations` if the table shape changed,
  then run `make sqlc`. Never write raw SQL inline in `db/actordelivery` Go
  code.
- `LeaseNextMailboxMessage` and `PeekNextMailboxMessage` share the same
  eligibility predicate and ORDER BY (priority DESC, available_at ASC,
  created_at ASC, with a per-correlation-key anti-join for FIFO lanes) — see
  the doc comments on both in `querier.go`. If one changes, the other must
  change to match, or the leaseless single-worker consume path
  (`db/actordelivery`) will silently diverge from the leased multi-worker
  path.
